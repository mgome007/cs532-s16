<article id="post-108" class="post-108 post type-post status-publish format-standard hentry category-uncategorized">
	
	<header class="entry-header">
		<h2 class="entry-title"><a href="http://devbrainhack.co/index.php/2015/11/09/recent-work-file-signatures-with-sindresorhusfile-type/" rel="bookmark">Recent work: sindresorhus/file-type and hex skills</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p>Yesterday my pull request for <a href="https://github.com/sindresorhus/file-type">sindresorhus/file-type</a> was <a href="https://github.com/sindresorhus/file-type/commit/17911e2de88050ce775fbf952ad2a84bd623b6dc">accepted</a>.  Hoorah! 😃</p>
<p><a href="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-5.34.24-PM.png"><img class="aligncenter wp-image-129 size-full" src="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-5.34.24-PM-e1447029449676.png" alt="Screen Shot 2015-11-08 at 5.34.24 PM" width="870" height="54" /></a></p>
<p><strong>I&#8217;ve decided to spend more time working with OSS developers for fun and profit. <a href="http://twitter.com/sindresorhus">Sindre Sorhus</a> is hugely popular as a developer, in large part for the the exceedingly numerous, high-quality packages he&#8217;s submitted to <a href="https://www.npmjs.com/~sindresorhus">NPM.org</a> and on <a href="http://github.com/sindresorhus">Github</a>.  All in all, he&#8217;s a stand-up guy who deserves the success and recognition for all the hard work he&#8217;s put in for the JS community.</strong></p>
<p>Lo-and-behold, a few days ago the call for contributors was made!</p>
<p>&nbsp;</p>
<blockquote class="twitter-tweet tw-align-center" lang="en">
<p dir="ltr" lang="en">Here are the open issues on my repos in case anyone wants to help out by triaging/commenting/fixing/etc ♥ <a href="https://t.co/R7DkIvjTBV">https://t.co/R7DkIvjTBV</a></p>
<p>— Sindre Sorhus (@sindresorhus) <a href="https://twitter.com/sindresorhus/status/661573963252932608">November 3, 2015</a></p></blockquote>
<p><script src="//platform.twitter.com/widgets.js" async="" charset="utf-8"></script></p>
<p>I landed on <a href="https://github.com/sindresorhus/file-type/issues/22">this issue</a> made for expanding <a href="https://github.com/sindresorhus/file-type">sindresorhus/file-type</a> with more compressed file types.  <a href="https://github.com/sindresorhus/file-type">file-type</a> is a Node.js module made for apps needing to identify files by their actual binary signature, similar to <a href="https://github.com/threatstack/libmagic">threatstack/libmagic</a> for the C language.  It has  a straightforward and concise structure consisting of a series of <code>if</code> statements and a byte-by-byte examination of the current file stream, typically only a few bytes. Here&#8217;s an example of how it checks a file if it&#8217;s a GIF:</p>
<pre class="EnlighterJSRAW" data-enlighter-language="js">if (buf[0] === 0x47 &amp;&amp; buf[1] === 0x49 &amp;&amp; buf[2] === 0x46) {
    return {
        ext: 'gif',
        mime: 'image/gif'
    };
}</pre>
<h4>Digging in&#8230;</h4>
<p><a href="http://www.crackmes.de/">I&#8217;ve messed with hex dumps and debugging binary files before</a>, and this simply involved cross-referencing the first few bytes of a file with already-known signatures.</p>
<p>The first  task was to get the <a href="https://github.com/sindresorhus/compressed-extensions/blob/master/compressed-extensions.json">lists of file types</a> <a href="https://github.com/sindresorhus/archive-extensions/blob/master/archive-extensions.json">needing signatures</a> and grabbing the relevant data from sources around the net, mostly from the <a href="http://www.garykessler.net/library/file_sigs.html">File Signatures Table, compiled by Gary Kessler</a>.</p>
<p style="text-align: justify;"><div class='content-column one_half'>I created a quick spreadsheet to keep things in order, too.</div><div class='content-column one_half last_column'><a href="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-5.32.45-PM.png"><img class="aligncenter size-large wp-image-127" src="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-5.32.45-PM-1024x507.png" alt="Screen Shot 2015-11-08 at 5.32.45 PM" width="660" height="327" /></a></div><div class='clear_column'></div>
<p>To help out on quick inspections, I created tiny utility to help at first, calling it fsig</p>
<pre class="EnlighterJSRAW" data-enlighter-language="shell">#!/bin/bash

clear
xxd $1 | head</pre>
<p>And for inspecting temporary directories of common file types, I used this bash one-liner:</p>
<pre class="EnlighterJSRAW" data-enlighter-language="null">clear; for n in *; do echo $n ; xxd $n | head ; done</pre>
<p>&nbsp;</p>
<p style="text-align: justify;"><div class='content-column one_half'>For extracting signatures cleanly, I ran into <a href="http://ridiculousfish.com/hexfiend/">HexFiend</a> for OSX.  It&#8217;s highly recommended since it displays where the cursor is by byte and allows for clean copy-paste.</div>
<div class='content-column one_half last_column'><a href="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-4.10.52-PM.png"><img class="aligncenter size-full wp-image-113" src="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-4.10.52-PM.png" alt="Screen Shot 2015-11-08 at 4.10.52 PM" width="806" height="886" /></a></div><div class='clear_column'></div>
<h4>Making it work</h4>
<p>A couple of problems I ran into was the issue of files with similar signatures.  For example, <code>deb</code> files are nearly identical to files made with the <code>compress</code> utility.  That means that care was needed when ordering the logic.</p>
<figure id="attachment_137" style="width: 660px;" class="wp-caption aligncenter"><a href="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-6.07.39-PM.png"><img class="wp-image-137 size-large" src="http://devbrainhack.co/wp-content/uploads/2015/11/Screen-Shot-2015-11-08-at-6.07.39-PM-1024x267.png" alt="ordering logic" width="660" height="172" /></a><figcaption class="wp-caption-text">Care was needed in ordering the logic for comparing file signatures</figcaption></figure>
<p>To make converting the signature hex into code simple and quick, I used Vim&#8217;s handy macro recording mode.<br />
<a href="http://devbrainhack.co/wp-content/uploads/2015/11/macro.gif"><img class="aligncenter size-full wp-image-139" src="http://devbrainhack.co/wp-content/uploads/2015/11/macro.gif" alt="macro" width="672" height="378" /></a></p>
<p><strong>I love Vim macros. </strong>Check out how to use them with <a href="http://usevim.com/2012/08/10/macros/">Vim 101: A Gentle Introduction to Macros</a>.</p>
<h4>All set</h4>
<p>Check out<a href="https://github.com/sindresorhus/file-type">sindresorhus/file-type</a>, and for that matter, maybe you can help with OSS by working on <a href="https://t.co/R7DkIvjTBV">open issues from sindresorhus&#8217; other projects</a> 😁</p>
<p>Let me know what you think on twitter <i class="fa fa-twitter"></i> <a href="http://twitter.com/lintuxvi">@lintuxvi</a>!</p>

	<div class="entry-content">
		<p>For the <a href="http://nbcuhackathon.com/en/orlando-2015/">2015 NBCU hackathon in Orlando</a>, I entered, along with a friend, into the Telemundo category.  We whipped up the design and the code in a night of a rough prototype app that outlined a novel way to generated viewer interaction during broadcasts.  Enter <a href="#" onClick="window.open('http://devbrainhack.co/projects/telemira/','_blank','toolbar=0, location=0, directories=0, status=0, scrollbars=0, resizable=0, copyhistory=0, menuBar=0, width=400, height=700');return(false)">TeleMira</a>:</p>
<figure id="attachment_74" style="width: 800px;" class="wp-caption aligncenter"><img class="wp-image-74 size-full" src="http://devbrainhack.co/wp-content/uploads/2015/11/telemira-e1446494862735.png" alt="TeleMira app" width="800" height="130" /><figcaption class="wp-caption-text">Introducing: TeleMira!</figcaption></figure>

<article id="post-48" class="post-48 post type-post status-publish format-standard hentry category-uncategorized tag-work">
	
	<header class="entry-header">
		<h2 class="entry-title"><a href="http://devbrainhack.co/index.php/2015/11/02/recent-work-bdd-with-angularjs-and-protractor/" rel="bookmark">Recent work: BDD with AngularJS and Protractor</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p>Recently, I spent a few days getting to know the testing ecosystem around AngularJS, specifically related to Unit Testing and End-to-End testing.  <a href="https://github.com/jasmine/jasmine">Jasmine</a> (unit-testing) and <a href="http://www.protractortest.org">Protractor</a> (end-to-end) are the two prescribed frameworks for those.</p>
<p><strong>To get a grasp of using these tools, I built a mock application: <a href="https://github.com/alexanderlperez/mailist">Mailist</a>.  Mailist&#8217;s story is that it helps send many emails easier, using a template and a <a href="https://en.wikipedia.org/wiki/Comma-separated_values">CSV</a> with recipient data.</strong></p>
<p>With that in mind, I looked around for some resources online as to how BDD changes the development process.  The first article to peak my interest was <em><a href="http://nicholasjackson.github.io/agile/bdd/angular/protractor/cucumber/protractor-cucumber/">Functional testing with cucumber and protractor</a>.  </em>Being that BDD differs from TDD in that a feature/<strong>behavior</strong> is being tested, rather than the functionality of a module/<strong>unit</strong>, bridging the conceptual gap was a little shaky.  In the middle of the <em><a href="http://nicholasjackson.github.io/agile/bdd/angular/protractor/cucumber/protractor-cucumber/">Functional testing&#8230;</a></em>, this graphic came up that helped to elucidate the process:</p>
<figure id="attachment_49" style="width: 681px;" class="wp-caption alignnone"><a href="http://devbrainhack.co/wp-content/uploads/2015/10/outside-in-development.png"><img id="figure-1" class="wp-image-49 size-full" src="http://devbrainhack.co/wp-content/uploads/2015/10/outside-in-development.png" alt="outside-in-development" width="681" height="332" /></a><figcaption class="wp-caption-text">Figure 1: the BDD cycle</figcaption></figure>
<p>Take a look at the graphic: you can see that unit-testing still exists within the context of the BDD, but now it&#8217;s involved within the development of the system at large.  Red-Green-Refactor is still there, but within the greater BDD loop of focusing on the <strong>scenario</strong>.</p>
<h4>Testing: BDD  👉  Scenarios</h4>
<p><a href="http://dannorth.net/whats-in-a-story/">Whats in a Story</a> is a great article that focuses on the scenario approach to writing features in BDD.  Boiled down, when we write out what the features will be:</p>
<h4>1. The scenario title should say what’s different</h4>
<blockquote><p>You should be able to line up the scenarios side by side, and describe how they differ using only the title.</p></blockquote>
<h4>2. The scenario should be described in terms of Givens, Events and Outcomes</h4>
<blockquote><p>&#8230;adopt this vocabulary of “given/when/then”&#8230;</p></blockquote>
<p>To build out the scenarios in <a href="https://github.com/alexanderlperez/mailist">Mailist</a>, these rules were used.  Here&#8217;s an example scenario:</p>
<pre class="toolbar:2 striped:false marking:false ranges:false nums:false nums-toggle:false wrap-toggle:false lang:default decode:true ">Scenario: A user loads a list and generates a queue
Given there are no emails
 And there is no template
 And there is no cached data
When the user loads a csv of data
Then the queue list should be populated
 And there should be relevant field headers</pre>
<p><span style="font-size: smaller;">Figure 2: example scenario</span></p>
<p>Note: this closely resembles the <a href="https://github.com/cucumber/cucumber/wiki/Gherkin">Gherkin</a> syntax, which I&#8217;d like to use with the <a href="https://github.com/cucumber/cucumber-js">Cucumber</a> BDD framework in the future, however I&#8217;m keeping this exploration simple.</p>
<h4>Testing: Protractor  👉  End-to-End</h4>
<p>Looking at the chart in Figure 1, since the features we build involve multiple parts of the system, testing at the <a href="https://en.wikipedia.org/wiki/Integration_testing">integration level </a>makes sense.  The Angular team has done a great job providing a tool just for that: <a href="http://www.protractortest.org">Protractor</a>.  It&#8217;s compatible with the Jasmine syntax, and uses a convenient element selection syntax for performing actions on target elements.  For example:</p>
<pre class="EnlighterJSRAW" data-enlighter-language="js" data-enlighter-linenumbers="false">var input = element(by.css('input'));
input.sendKeys('123');
expect(input.getAttribute('value')).toBe('Foo123');</pre>
<p>Check out <a href="http://www.protractortest.org/#/api?view=ElementFinder">the docs</a> for more info.</p>
<h4>Testing: Jasmine  👉 Unit-Testing</h4>
<p>Here we unit test with the popular <a href="http://www.protractortest.org">Jasmine test framework</a>.  For each of the scenarios, now a series of internal modules become clear.  <strong>This was the aha moment.</strong>  <em>For each scenario, the internal requirements become clear.</em>  And just like the chart in Figure 1, I&#8217;m back to the Red-Green-Refactor cycle, defining and testing module behavior in the typical TDD fashion.</p>
<p>The one thing that sticks out here with Angular is testing directives.  There are a couple of methods I&#8217;ve seen online recommended: using E2E for testing directives, or using the <a href="https://docs.angularjs.org/guide/unit-testing#testing-directives">internal angular directive compilation</a>.  Both are straightforward, however I was having trouble automating the file input through the directive compilation method because of browser security policies.  I&#8217;d rather not have to script launching browsers with special access policy flags, so I left those tests to Protractor.<b></b></p>
<h4>Conclusion</h4>
<p>So, in short, BDD with Angular is straightforward using Protractor and Jasmine for E2E and unit-testing.  By sticking to the chart in <a href="#figure-1">Figure 1</a>, building out scenarios, and sticking with givens/events/outcomes to outline our scenarios, we have a clear idea of how our app evolves.</p>
<h4>Links</h4>
<p>Jump over to Github and <a href="https://github.com/alexanderlperez/mailist">browse the code for Mailist</a>.  Then take a look at these links for the tools mentioned:</p>
<ul>
<li><a href="https://github.com/jasmine/jasmine">Jasmine</a></li>
<li><a href="http://www.protractortest.org">Protractor</a></li>
</ul>
<p>For my favorite link on Angular testing, check out Year of Moo&#8217;s <a href="http://www.yearofmoo.com/2013/01/full-spectrum-testing-with-angularjs-and-karma.html">Full-Spectrum Testing with AngularJS and Karma</a>.  It&#8217;s centered around Mocha, Chai, and Karma as a testing framework, but the concepts for testing each part of an Angular app is broken down here.</p>
<p>Let me know what you think on Twitter <a href="http://www.twitter.com/lintuxvi">@lintuxvi</a></p>

	<header class="entry-header">
		<h2 class="entry-title"><a href="http://devbrainhack.co/index.php/2015/09/02/baking-partnerships/" rel="bookmark">Baking partnerships</a></h2>	</header><!-- .entry-header -->

	<div class="entry-content">
		<p><span style="font-size: 2em; font-style: bold;">I</span>&#8216;ve been wanting to write something for myself for a while now, and being a tech blog, I should inaugurate this one with an instance of writing blessed by tech canon: fail often and fail quick.  Figuratively, you can see it like playing a poker hand or snipping a wire defusing a bomb.  As far as consequences go, they can range from emotionally draining to traumatizing (in a sort of gradient between n&#8230;n+1).  But if we&#8217;re torturing analogies here, let&#8217;s talk about baking and partnerships.</p>
<p>I&#8217;m mostly a coder, who dabbles in entrepreneurship, but since my passion lies in personal projects and friendships, I&#8217;ve run into a few situations where they come together.  Some of these Frankensteins have turned out well, and others not so much, but the biggest factor in determining that success has been in the partnership built.</p>
<h4>Picking a partner: values and experience aren&#8217;t all sugar and spice</h4>
<p>Picking a partner is a kind of baking.  You get the ingredients together, you lay them out in front of you, then crack, crack, crack, mix, mix, mix, whisk them all in a bowl and stick it in an oven.  Your project, the business, is like that oven.</p>
<p>Cranking the oven up reveals the wonderful possibilities, and without it your cake is just a pan of half-rendered goop, in a state of ready decomposition.  By taking the elements and putting them to the test, all the constituent elements behave together as well as the mixed ingredients were picked and placed.</p>
<p><a href="http://devbrainhack.co/wp-content/uploads/2015/09/herbs-and-spices.jpg"><img class="size-medium wp-image-20 aligncenter" src="http://devbrainhack.co/wp-content/uploads/2015/09/herbs-and-spices-300x210.jpg" alt="herbs-and-spices" width="300" height="210" /></a></p>
<p>Except, <em>sometimes</em>, we as bakers are blind.  We as bakers mislabel our ingredients, or ignore the labels altogether.  What was salt is actually sugar and what is sugar is baking soda, and what is baking soda is Ajax scrubbing powder with bleach.  Yech, and call your local poison-control center immediately.</p>
<p>People are the same.  You and I, we hold a few flavors for public consumption.  How we behave, the attitudes we project, our go-to methods for conflict resolution and problem-solving: we keep these as a sort of feature-set called upon as long as there&#8217;s clean, un-pressured energy to utilize before our fight-or-flight centers kick in.</p>
<p>Take multiple people, add a potentially big payout, and now we&#8217;ve got some heat.  You&#8217;re going to run into situations when what you thought was a Beef-Wellington ends up being just a starter-log wrapped in yolk-glazed paper-mache.  Volatile, flammable.</p>
<p>Some have ingredients rated for high temperatures, while others go up in flames pretty quickly.</p>
<p>So, big note-to-self: <span style="text-decoration: underline;">put the time in beforehand.</span></p>
<ol>
<li>&#8216;Date&#8217; all potential partners, and be aware of the typical warning signs (ex. mismatched values, gut feelings, personal history)</li>
<li>Make sure the vision is there.  No vision is like a train off the rails: crazy.</li>
<li>Make sure there&#8217;s an end-game. Get confident about it, learn the possibilities and contingencies.</li>
<li>Have an understanding of how each partner resolves uncertainty and conflict.  One person making decisions out of fear in a partnership is a symptom that real communication has broken down and planning is out the window.  That&#8217;s a recipe for burnout and disaster.</li>
<li>Make sure negotiating financials isn&#8217;t akin to pulling teeth.  Someone who continually tries to negotiate you at every step is a sign that you&#8217;re not going to be working together long.</li>
</ol>
<p>If you&#8217;re going to be spending irretrievable time in your life with others, with the intentional goal of creating something to benefit everyone involved, make sure you all balance each other&#8217;s strengths and weaknesses in a lucid and explicit way, even at the worst of times.<strong><em>  And that&#8217;s not easy to achieve.  </em></strong>But putting in the legwork beforehand will be worth it in the end.</p>
	</div><!-- .entry-content -->
