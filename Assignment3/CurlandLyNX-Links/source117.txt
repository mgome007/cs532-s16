          <div class="date-posts">

<div class='post hentry' itemscope='itemscope' itemtype='http://schema.org/BlogPosting'>

<h3 class='post-title entry-title' itemprop='name'>
<a href='http://www.runtime-era.blogspot.com/2015/04/solid-review-dependency-inversion.html'>SOLID Review: Dependency Inversion Principle</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-2346769048995641211' itemprop='articleBody'>
<div class="separator" style="clear: both; text-align: center;"><a href="http://3.bp.blogspot.com/-q2ANPJs05Cw/VRUHZ4f-QJI/AAAAAAAAEY0/bmkynm7qZ9c/s1600/plug.jpg" imageanchor="1" style="margin-left: 1em; margin-right: 1em;"><img border="0" src="http://3.bp.blogspot.com/-q2ANPJs05Cw/VRUHZ4f-QJI/AAAAAAAAEY0/bmkynm7qZ9c/s320/plug.jpg" /></a></div>
<i>Note: This is part of a series of articles reviewing the <a href="http://en.wikipedia.org/wiki/SOLID_%28object-oriented_design%29">five SOLID Principles of object-oriented programming</a>.</i>

<br/>
<br/>

The final SOLID principle is known as the Dependency Inversion principle. Arguably the most important of the five principles, the Dependency Inversion principle can be thought of as a culmination of the principles preceding it. Systems that abide by the other SOLID principles tend to follow the Dependency Inversion principle as a result. The principle states:
<br/>
<br/>

<b>"High-level modules should not depend on low-level modules."</b>
<br/>
<br/>

A better way to think about it is:
<br/>
<br/>

<b>"Abstractions should not depend upon details. Details should depend upon abstractions."</b>
<br/>
<br/>

In a static-typed language like Java, "abstractions" can be implemented and enforced explicitly via <a href="http://docs.oracle.com/javase/tutorial/java/concepts/interface.html">interfaces</a>. However, in a dynamic language like Ruby, we depend on <a href="http://en.wikipedia.org/wiki/Duck_typing">duck-typing</a> to describe an object's interface. Even without explicit interfaces in Ruby, the Dependency Inversion principle still holds value! We should still aim to <b>depend on abstractions rather than details</b>.
<br/>
<br/>

Let's look at an example! We'll revisit a simple example from my <a href="http://www.runtime-era.com/2015/02/solid-review-openclosed-principle.html">blog post on the Open/Closed Principle</a>.
<br/>
<br/>

<h3>A Simple String Transformer</h3>
Suppose we have a class called <b>Transformer</b> that takes a string and transforms it into a some other object or value. For starters, we'll have it transform JSON strings into Ruby hashes:

<pre class="brush: ruby">
require 'json'

class Transformer
  def initialize(string)
    @string = string
  end

  def transformed_string
    JSON.parse(@string)
  end
end

Transformer.new('{"foo": "bar"}').transformed_string
# { "foo" => "bar" }

</pre>

Now, we'll extend the functionality of our Transformer by allowing it to transform strings into binary:

<pre class="brush: ruby">
require 'json'

class Transformer
  def initialize(string)
    @string = string
  end

  def transformed_string(type)
    if type == :json
      JSON.parse(@string)
    elsif type == :binary
      @string.unpack('B*').first
    end
  end
end

Transformer.new('Hello').transformed_string(:binary)
# "0100100001100101011011000110110001101111"
</pre>

Now, our Transformer takes strings and transforms them into one of two different types: a Ruby hash or its binary representation. At this point, we should notice some code-smell! The <b>transformed_string</b> method is very dependent on <b>JSON.parse</b> and <b>String.unpack</b>. These are implementation details that our Transformer shouldn't care about. 
<br/>
<br/>

Let's apply the Dependency Inversion principle by making Transformer depend on <i>an abstraction</i> rather than coupling to concrete details!
<br/>
<br/>

<h3>The Transformation Abstraction</h3>
The basic functionality of our Transformer class is to transform strings into several different types of objects or values. It does this by utilizing different <i>transformations</i>. This seems like an abstraction we can extract and encapsulate! We'll make Transformer depend on a new abstraction called <b>Transformation</b>:

<pre class="brush: ruby">
class Transformer
  def initialize(string)
    @string = string
  end

  def transformed_string(transformation)
    transformation.transform(string)
  end
end

class BinaryTransformation
  def self.transform(string)
    string.unpack('B*').first
  end
end

Transformer.new('Hello').transformed_string(BinaryTransformation)
# "0100100001100101011011000110110001101111"

require 'json'

class JSONTransformation
  def self.transform(string)
    JSON.parse(string)
  end
end

Transformer.new('{"foo": "bar"}').transformed_string(JSONTransformation)
# { "foo" => "bar" }
</pre>

Rather than having Transformer depend on low-level implementation details (JSON.parse and String.unpack), it now depends on a single method: <b>transform</b>. This single method is what makes up the interface of our Transformation abstraction! Now, we can create as many Transformations as we want without modifying Transformer:

<pre class="brush: ruby">
require 'digest'

class MD5Transformation
  def self.transform(string)
    Digest::MD5.hexdigest string
  end
end

Transformer.new('Hello').transformed_string(MD5Transformation)
# "8b1a9953c4611296a827abf8c47804d7"
</pre>


<h3>Conclusion</h3>
As you can see, the <a href="http://www.runtime-era.com/2015/02/solid-review-openclosed-principle.html">Open/Closed principle</a> is highly correlated with the Dependency Inversion principle! We actually end up following the Open/Closed principle by abiding by the Dependency Inversion principle. In fact, some form of dependency abstraction is often required to abide by all the other SOLID principles. If there's one principle to remember out of all the SOLID principles, it's the Dependency Inversion principle: <b>depend on abstractions, not low-level details</b>!
<br/>
<br/>

Happy coding!

          <div class="date-posts">

<h3 class='post-title entry-title' itemprop='name'>
<a href='http://www.runtime-era.blogspot.com/2015/03/solid-review-interface-segregation.html'>SOLID Review: Interface Segregation Principle</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-4174968216245608109' itemprop='articleBody'>
<div class="separator" style="clear: both; text-align: center;"><a href="http://3.bp.blogspot.com/-GrMpVTN0_ZY/VNVXx74V5xI/AAAAAAAAD3U/t-LPYfNmiZM/s1600/clutter.jpg" imageanchor="1" style="margin-left: 1em; margin-right: 1em;"><img border="0" src="http://3.bp.blogspot.com/-GrMpVTN0_ZY/VNVXx74V5xI/AAAAAAAAD3U/t-LPYfNmiZM/s320/clutter.jpg" /></a></div>
<br/>

<i>Note: This is part of a series of articles reviewing the <a href="http://en.wikipedia.org/wiki/SOLID_%28object-oriented_design%29">five SOLID Principles of object-oriented programming</a>.</i>
<br/>
<br/>

The Interface Segregation Principle is probably the most straight-forward of all the SOLID principles. It states:
<br/>
<br/>

<b>"Clients should not be forced to depend on methods that they do not use."</b>
<br/>
<br/>

In dynamic languages, this isn't really much of an issue because there is no way to define and force the implementation of interfaces on classes (<a href="http://docs.oracle.com/javase/tutorial/java/concepts/interface.html">like in Java</a>). Instead, <b>a set of methods</b> determines whether or not an object implements an interface. If an object responds to "a particular set of methods", it has implemented that "particular interface".
<br/>
<br/>

In Ruby, <a href="http://www.ruby-doc.org/core-2.2.0/Module.html">modules</a> can be used to define and share sets of methods across multiple classes. Using this construct, we can define different "interfaces". So, when we say "keep our interfaces segregated", we're really saying "keep our modules segregated". This leads to highly cohesive modules.
<br/>
<br/>

There are two main benefits to cohesive modules in Ruby: <b>less coupling</b> and <b>more readable code</b>.
<br/>
<br/>

<h3>Implementing Phones</h3>
By keeping our modules small and focused, we are simply applying the <a href="http://www.runtime-era.com/2015/02/solid-review-single-responsibility.html">Single Responsibility Principle</a>, but for modules. For example, let's create a module called <b>Phone</b>:

<pre class="brush: ruby">
module Phone
  def call(number)
    "Calling #{number}..."
  end

  def hangup
    "Hanging up!"
  end

  def text(number, message)
    "Texting '#{message}' to #{number}."
  end
end
</pre>

Here, we have a set of common behaviors for phones. We can make use of this by including them in our class. Let's create a <b>CellPhone</b>:

<pre class="brush: ruby">
class CellPhone
  include Phone
end
</pre>

Now, our CellPhone class implements the methods in Phone! Any instance of CellPhone can <b>call</b>, <b>hangup</b>, and <b>text</b> other numbers.
<br/>
<br/>

Let's create a new class called <b>RotaryPhone</b>:

<pre class="brush: ruby">
class RotaryPhone
  include Phone

  # Eek... code smell.
  def text(number, message)
    raise 'Cannot text on this type of phone!'
  end
end
</pre>

Since we are overriding one of the methods in our module, it's a sign our module isn't cohesive enough. Our RotaryPhone is being littered with methods it does't need! 
<br/>
<br/>

Another issue worth noting is the tight coupling between our two classes caused by sharing the same, non-cohesive module. Suppose we don't override the <b>text</b> method in RotaryPhone:

<pre class="brush: ruby">
class RotaryPhone
  include Phone
end
</pre>

Any errors caused by <b>text</b> in our Phone module would end up in both classes, even though RotaryPhone doesn't care about <b>text</b>! This tight coupling between CellPhone and RotaryPhone is unnecessary.
<br/>
<br/>

<h3>Segregate the Modules</h3>
A good solution for our problem is to <i>segregate</i> the basic phone behaviors from the mobile phone behaviors:

<pre class="brush: ruby">
module BasicPhone
  def call(number)
    "Calling #{number}..."
  end

  def hangup
    "Hanging up!"
  end
end

module MobilePhone
  def text(number, message)
    "Texting '#{message}' to #{number}."
  end
end
</pre>

Now, each of our classes implement only the modules they require:

<pre class="brush: ruby">
class CellPhone
  include BasicPhone
  include MobilePhone
end

class RotaryPhone
  include BasicPhone
end
</pre>

<h3>A Readable, Loosely-Coupled Solution</h3>
The behaviors of each class are more clearly defined by the explicitness of the modules it includes. Also, CellPhone and RotaryPhone are only coupled by the methods in BasicPhone, which makes sense since they both require the basic behaviors or <b>call</b> and <b>hangup</b>. Both of our issues above are solved!
<br/>
<br/>

<h3>Conclusion</h3>
Although the Interface Segregation Principle is less important in dynamic languages like Ruby, it still leads to cohesive, readable classes. By keeping modules focused, we end up with looser coupling and cleaner "interface" definitions. They aren't major wins, but wins nonetheless!
<br/>
<br/>

Happy coding!

          <div class="date-posts">
        
<div class='post-outer'>
<div class='post hentry' itemscope='itemscope' itemtype='http://schema.org/BlogPosting'>
<a name='4716458437784896040'></a>
<h3 class='post-title entry-title' itemprop='name'>
<a href='http://www.runtime-era.blogspot.com/2015/03/solid-review-liskov-substitution.html'>SOLID Review: Liskov Substitution Principle</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-4716458437784896040' itemprop='articleBody'>
<div class="separator" style="clear: both; text-align: center;"><a href="http://4.bp.blogspot.com/-e6lQpjuEvRY/VNVWCkSx8CI/AAAAAAAAD3I/nja2Io84Ufk/s1600/duck.jpg" imageanchor="1" style="margin-left: 1em; margin-right: 1em;"><img border="0" src="http://4.bp.blogspot.com/-e6lQpjuEvRY/VNVWCkSx8CI/AAAAAAAAD3I/nja2Io84Ufk/s320/duck.jpg" /></a></div>
<br/>

<i>Note: This is part of a series of articles reviewing the <a href="http://en.wikipedia.org/wiki/SOLID_%28object-oriented_design%29">five SOLID Principles of object-oriented programming</a>.</i>
<br/>
<br/>

Barbara Liskov introduced her substitution principle back in 1987 during her keynote titled <a href="http://rendezvouswithdestiny.net/index_files/LiskovSub.pdf">Data Abstraction and Heirarchy</a>. Today, it is one of the five SOLID principles in object-oriented programming. The original definition is as follows:
<br/>
<br/>

<b>"Let q(x) be a property provable about objects x of type T. Then q(y) is provable for objects y of type S, where S is a subtype of T."</b>
<br/>
<br/>

Simply put:
<br/>
<br/>

<b>"Instances of any type should be replaceable by instances of its subtypes without creating incorrect behaviors."</b>
<br/>
<br/>

How can we ensure that our classes abide by the Liskov Substitution Principle? For starters, we must ensure that any subtype implements the interface of its base type. In the world of dynamic languages, this is better stated as <b>a subtype must <i>respond to</i> the same set of methods as its base type</b>.
<br/>
<br/>

We must also ensure that methods in any subtype <b>preserve the original <i>promises</i> of methods in its base type</b>. What "promises" are we talking about? For that, we turn to another design principle known as <b>Design by Contract</b>.
<br/>
<br/>

<h3>Design by Contract</h3>
The concept of <a href="http://en.wikipedia.org/wiki/Design_by_contract">Design by Contract</a> was coined by Bertrand Meyer in his book <a href="http://www.amazon.com/Object-Oriented-Software-Construction-CD-ROM-Edition/dp/0136291554">Object Oriented Software Construction</a>. It's official description is much more detailed, but to paraphrase, there are three basic principles:

<ul>
  <li>Subtypes <b>should not strengthen any preconditions</b> of its base type. That is, requirements on inputs to a subtype cannot be stricter than in the base type.</li>
  <li>Subtypes <b>should not weaken any postconditions</b> of its base type. That is, the possible outputs from a subtype must be more than or equally restrictive as from the base class.</li>
  <li>Subtypes must <b>preserve all invariants</b> of its base type. That is, if the base type has guarantees that certain conditions be true, its subtype should make those same guarantees.</li>
</ul>
<br/>

If any of the above are violated, chances are the Liskov Substitution Principle is also violated.
<br/>
<br/>

<h3>A Liskov Substitution Checklist</h3>
Let's look at a simple example. We're going to model several types of birds. We'll start by defining a base type called <b>Bird</b>:

<pre class="brush: ruby">
class Bird
  def initialize
    @flying = false
  end

  def eat(food)
    if ['worm', 'seed'].include?(food)
      "Ate #{food}!"
    else
      raise "Does not eat #{food}!"
    end
  end

  def lay_egg
    # The Egg class has a method 'hatch!' that returns a new Bird.
    Egg.new 
  end

  def fly!
    @flying = true
  end
end
</pre>

Instances of Bird are very simple. They eat only certain types of food, lay eggs, and can go from sitting on the ground to flying in the air. For now, ignore the fact that our Bird cannot go back on the ground. Here's a small program that uses our Bird:

<pre class="brush: ruby">
bird = Bird.new

bird.eat('worm') # Ate worm!

egg = bird.lay_egg # Returns an Egg
egg.hatch! # Returns a new Bird

bird.fly! # @flying == true
</pre>

Remember, any subtypes from Bird should be able to work in our program above. Now, let's create some subtypes of Bird and see how we can apply the Liskov Substitution Principle.
<br/>
<br/>

<ul style="list-style-type: square;">
<li><h3>The subtype must implement the base type's interface.</h3></li>
</ul>

In most programming languages, we can achieve this through basic inheritance. Since we already have a base class defined, we'll take this approach. However, there are many ways to achieve this across many languages. In Ruby, we can use <a href="http://www.ruby-doc.org/core-2.2.0/Module.html">modules</a> to share methods (see <a href="http://en.wikipedia.org/wiki/Duck_typing">duck-typing</a>). In Java, we can implement <a href="http://docs.oracle.com/javase/tutorial/java/concepts/interface.html">interfaces</a>.
<br/>
<br/>

Let's create a <b>Pigeon</b> subclass:

<pre class="brush: ruby">
class Pigeon < Bird
end

bird = Pigeon.new # Behaves exactly like Bird!
</pre>

Success! Pigeon now implements Bird's interface.
<br/>
<br/>

<ul style="list-style-type: square;">
<li><h3>The subtype should not strengthen preconditions of the base type.</h3></li>
</ul>

Let's say our Pigeons can only eat bread. We will override the <b>eat</b> method to achieve this:

<pre class="brush: ruby">
class Pigeon < Bird
  def eat(food)
    if ['bread'].include?(food)
      "Ate #{food}!"
    else
      raise "Does not eat #{food}!"
    end
  end
end

# bird is now Pigeon
bird.eat('worm') # raises an error: "Does not eat worm!"
</pre>

Since we've actually <b>made the preconditions to our method <i>stricter</i></b> than in the Bird class, we've violated the Liskov Substitution Principle! In doing so, we've broken our existing program!
<br/>
<br/>

Instead, let's say that Pigeons can eat bread <i>in addition</i> to seeds and worms. Then, we've <i>weakened</i> the preconditions and are well within our rule:

<pre class="brush: ruby">
class Pigeon < Bird
  def eat(food)
    if ['worm', 'seed', 'bread'].include?(food)
      "Ate #{food}!"
    else
      raise "Does not eat #{food}!"
    end
  end
end

bird.eat('worm') # "Ate worm!"
</pre>

And our program works with our subclass!
<br/>
<br/>

<ul style="list-style-type: square;">
<li><h3>The subtype should not weaken postconditions of the base type.</h3></li>
</ul>

Let's say our Pigeon is some kind of mutant and doesn't actually lay eggs. We'll call it a <b>MutantPigeon</b>. Instead, no egg comes out at all:

<pre class="brush: ruby">
class MutantPigeon < Bird
  def lay_egg
    nil
  end
end

bird = MutantPigeon.new

egg = bird.lay_egg # returns nil
egg.hatch! # raises an error: undefined method 'hatch!' for nil:NilClass
</pre>

We've broken our program yet again! Since we've actually <b>made the postconditions in our method <i>less restrictive</i></b> than in the Bird class, we've violated the Liskov Substitution Principle.
<br/>
<br/>

Instead, let's say that MutantPigeons actually return a <i>more specific</i> type of Egg. We'll call it <b>MutantPigeonEgg</b>, and it behaves just like Egg with a <b>hatch!</b> method. Then, we've <i>strengthened</i> the postconditions and are well within our rule:

<pre class="brush: ruby">
class MutantPigeon < Bird
  def lay_egg
    MutantPigeonEgg.new
  end
end

egg = bird.lay_egg # returns nil
egg.hatch! # Returns a new MutantPigeon
</pre>

And our program is happy again!
<br/>
<br/>

<ul style="list-style-type: square;">
<li><h3>The subtype should preserve invariants of the base type.</h3></li>
</ul>

Let's model a different bird this time. What about <b>Penguin</b>s? As many people know, most penguins in the real world don't actually fly. So, we'll override the <b>fly</b> method with a no-op:

<pre class="brush: ruby">
class Penguin < Bird
  def fly
    # no-op, do nothing
  end
end

bird = Penguin.new

bird.fly! # @flying != true
</pre>

Looks like another break in our program! By doing nothing in our new <b>fly</b> method, we've <b><i>broken</i> the guarantee</b> that the state of our <b>@flying</b> variable would be "true". Again, we've violated the Liskov Substitution Principle.
<br/>
<br/>

Now, this introduces an interesting problem. Penguins cannot just be made to fly, right?!
<br/>
<br/>

<h3>Real-Life Relationships != Inheritance-Model Relationships</h3>
Objects in the real world may show an obvious inheritance relationship. However, in object-oriented design, we only care about inheritance relationships regarding object <i>behavior</i>. Think of the classes in our system as <b>representations</b> of real-world objects. Those representations are fully defined by their external behavior (or interface). 
<br/>
<br/>

Sure, penguins are birds in the real world, but Penguins <i>are not</i> Birds in our system because they do not <i>behave</i> like Birds. They don't have a properly functioning <b>fly</b> method.
<br/>
<br/>

<h3>Liskov Substitution and the Open/Closed Principle</h3>
Consider the examples above. Suppose we actually violated the Liskov Substitution Principle by creating our Pigeon class with a more restrictive <b>eat</b> method? Our existing program would have to be modified to handle our new class:

<pre class="brush: ruby">
class Pigeon < Bird
  def eat(food)
    if ['bread'].include?(food)
      "Ate #{food}!"
    else
      raise "Does not eat #{food}!"
    end
  end
end

if bird.instance_of?(Pigeon)
  bird.eat('bread') # "Ate bread!"
else
  bird.eat('worm') # "Ate worm!"
end
</pre>

As we know from the Open/Closed Principle, we shouldn't have to change existing code to add new requirements or features. By violating the Liskov Substitution Principle, we are forced to violate the Open/Closed Principle!
<br/>
<br/>

<h3>Conclusion</h3>
As with all programming principles, it's important to find a balance when applying the Liskov Substitution Principle in  real-world scenarios. There is <a href="http://c2.com/cgi/wiki?LiskovSubstitutionPrinciple">some debate</a> over the benefits or detriments of the principle. Always keep it simple first, then refactor as needed.
<br/>
<br/>

Happy coding!
